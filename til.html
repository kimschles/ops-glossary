  • A Kubernetes volume is a directory that contains files. The files in a volume are available to any container in the pod that has a volume, and the files disappear when the pod is deleted, but stay when the pod is restarted. The files can be accessed at the volumeMount, which gets a name, and a mountPath, which provides a path to the directory which contains the files.
  • There are different types of Kubernetes volumes. For directories that are on the node, use emptyDir or hostPath. For cloud-provider volumes, use awsElasticBlockStore or gcePersistentDisk. If you want a distributed file system, use cephfs, and there are special types of volumes like secret. Everything I know about volumes I learned from Kubernetes volumes by example

  • LetsEncrypt verifies that you control the domain names you claim via a challenge. The two most common types of challenges are HTTP-01 (the recommended Let's Encrypt default) and the DNS-01. I the HTTP-01 challenge, Let's Encrypt gives a token to your ACME client, and the ACME client puts a file on your webserver that has the token and a print of your account key. Let's Encrypt tries to access that file in multiple different ways, and if it gets the response it expects, Let's Encrypt will issue a certificate. The DNS-01 challenge checks for a TXT record affiliated with the domain name. This is what I'm setting up when Firebase asks me to verify my domain. üòÅ A great resource: Let's Encrypt Challenge Types

  • In software, the term 'backport' means taking some lines of code from new software, and using it in an older version of the same software. This is often done to fix security issues in older software.
  • Audit logs (AKA an audit trail), are records of a change to a system. In Kubernetes, a way to figure audit logs is to record all `GET` `POST` `DELETE` and `PATCHES`, the entity making those requests (a K8s controller or a person, etc), and the result of that request. Audit logs are often looked at when examining a security breach, or analyzed to uncover trends that could be fixed. Some ways of looking at audit logs are Google's Stackdriver, ElasticSearch + Kibana, and Datadog. Audit logs can be _a lot_ of data, so setting up limits is likely a cost-saving measure.

  • A service is a Kubernetes object that has a stable IP address and port, accepts traffic from the internet and directs it to a set of pods. A service sits in front of a set of pods, and has a stable IP because pods are often killed and recreated with different IP addresses. In Kubernetes, there are three types of services: (Quoted material is from https://kubernetes.io/docs/concepts/services-networking/service/) - `clusterIp` (for internal use only) > Exposes the Service on a cluster-internal IP. Choosing this value makes the Service only reachable from within the cluster. This is the default `ServiceType`. - `NodePort` > Exposes the Service on each Node‚Äôs IP at a static port (the `NodePort`). A `ClusterIP` Service, to which the `NodePort` Service routes, is automatically created. You‚Äôll be able to contact the `NodePort` Service, from outside the cluster, by requesting `:`. - `LoadBalancer` > Exposes the Service externally using a cloud provider‚Äôs load balancer Don‚Äôt use `NodePort` for these reasons: - `NodePort` is bad for network security because you have to allow or disallow all traffic. - `NodePort` port ranges are from `30000-32767`, which is out of range for HTTPS (443), SSH (22) , and HTTP (80) ports. - If your Node/VM IP addresses change, your `NodePort` address will be broken `NodePort` is useful for demo apps or tutorials, not running things in production. At Fairwinds, we use an ingress controller to accept traffic from the internet, and then route it to the proper service. Resources: - https://oteemo.com/2017/12/12/think-nodeport-kubernetes/ - https://medium.com/google-cloud/kubernetes-nodeport-vs-loadbalancer-vs-ingress-when-should-i-use-what-922f010849e0
